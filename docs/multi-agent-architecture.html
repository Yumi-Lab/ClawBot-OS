Règle fondamentale — pas d'IA dans l'IA

Les agents ne peuvent pas spawner d'autres agents. chat_with_agent_stream() appelle chat_with_tools_stream(), jamais chat_with_multi_agents_stream(). Pas de récursion infinie possible.

Les 2 modes de chat actifs

BoutonEndpointFonction backendComportement
ClawBot Core /v1/chat/completions chat_with_tools_stream() Un seul LLM, tool loop complet, tous les outils disponibles
Core-Agent /v1/chat/agents chat_with_multi_agents_stream() Haiku route → agent sélectionné → tool loop avec skills filtrés

Mode Auto — supprimé. L'ancien agent Go tentait de se connecter à 148.163.48.215:443 (service cloud mort). Ne pas réactiver.

Le routeur Haiku est toujours Haiku, indépendamment du modèle sélectionné dans le dashboard. C'est intentionnel : classification rapide, cheap, max 30 tokens.

Comment le modèle est transmis aux agents

request_body contient model: "kimi-for-coding" (ou autre). Ce dict est passé tel quel à chat_with_agent_stream(request_body, agent_id) → puis à chat_with_tools_stream(body, override_tools) → puis à _call_llm_stream(body). Le modèle ne change jamais en cours de route.

Conclusion : sélectionner KimiCode 2.5 dans le dashboard = tous les agents travaillent avec KimiCode 2.5.

Routage automatique (Core-Agent)

Algorithme de routage

Message ≤ 3 mots ?
OUI →Core direct. Pas d'appel LLM routeur. Économie tokens.
NON →Appel Haiku (_route_via_llm)
Haiku retourne un agent_id ?
OUI →chat_with_agent_stream(request_body, agent_id) — agent sélectionné
"none" →Fallback Core — chat_with_tools_stream(data) — tous les outils, pas de persona
Haiku échoue (timeout, erreur) ?
ERREUR →_route_via_keywords() — matching par mots-clés (score), fallback keyword

Prompt du routeur Haiku

"Tu es un routeur. Décide si le message nécessite un agent spécialisé ou peut être traité directement.
Règle principale : réponds 'none' si :
- la conversation est simple, générale ou conversationnelle
- aucune compétence spécifique n'est clairement requise
Ne sélectionne un agent QUE si la tâche nécessite ses compétences précises.

Agents disponibles:
- id: python-dev | name: Python Dev | skills: system__bash, system__python | role: ...
- id: sysadmin | name: SysAdmin | skills: system__bash, system__ssh | role: ...
...

Message: "[user_message]"

Réponds UNIQUEMENT avec l'id de l'agent ou: none"

Haiku a max_tokens: 30 et timeout: 15s. Il retourne juste un mot.

Routing multi-agent (plusieurs agent_ids)

Si le frontend envoie agent_ids: ["sysadmin", "python-dev"], le routage automatique est bypassé et les deux agents tournent en parallèle via ThreadPoolExecutor.

    ├── ThreadPoolExecutor(max_workers=N)
    │   ├── Thread A → chat_with_agent_stream(body, "sysadmin")

Parallel tool calls dans un seul tour

Dans chat_with_tools_stream(), quand le LLM retourne plusieurs tool_calls dans une seule réponse (ex: Kimi K2 appelle bash + web_search en même temps), ils sont exécutés en parallèle :

if len(tool_calls) > 1:
    with ThreadPoolExecutor(max_workers=min(len(tool_calls), 4)) as ex:
        futures = {ex.submit(_run_tc_stream, tc): tc.get("id") for tc in tool_calls}
        for fut in as_completed(futures):
            tc_id, name, result = fut.result()

C'est la feature "parallelization" de Kimi K2 — il peut émettre plusieurs tool_calls en une passe. Notre système les exécute déjà en parallèle nativement.

Twins Partner (cycle de review)

chat_with_agent_stream()
    1. LLM génère réponse complète (full_response)
    2. Si twins_partner == true :
       a. Haiku évalue : [APPROVED] ou [NEEDS_REVISION] + feedback
       b. Si NEEDS_REVISION → nouvelle requête LLM avec feedback
       c. Retourne réponse révisée
    3. Yield event "done" final

Le reviewer Haiku est hardcodé et appelle Anthropic directement (pas via le cloud), avec sa propre clé API.

Haiku 4.5 en mode agent

Points forts

Très rapide (< 1s pour routing). Économique. Bon pour tâches simples, questions courtes, file operations.

Limites

Parallel tool calls moins agressifs. Raisonnement limité pour tâches multi-étapes complexes. Max 200k ctx.

Recommandé pour : usage quotidien, queries simples, économiser les tokens du plan.

Sonnet 4.6 en mode agent

Points forts

Bon équilibre vitesse/intelligence. Tool calling fiable. Génère du code de qualité. 200k ctx.

Limites

Plus lent que Haiku. Coût $3/$15 /M tokens. Pas de reasoning natif (pas de thinking visible).

Recommandé pour : développement, debugging, tâches techniques modérément complexes.

Opus 4.6 en mode agent

Points forts

Le plus puissant. Excellent pour planification complexe, décomposition de tâches, architecture système. 200k ctx.

Limites

Coût $15/$75 /M tokens. Lent. À utiliser avec parcimonie. Peut être overkill pour tâches simples.

Recommandé pour : refactoring majeur, architecture, tâches nécessitant un raisonnement multi-étapes profond.

KimiCode 2.5 en mode agent

Points forts

262k contexte (vs 200k Claude). Parallel tool calls natifs. Entraîné pour l'agentic coding. Budget économique. Reasoning intégré (reasoning_content).

Spécificités

Requiert User-Agent: claude-code/1.0. Appelé via cloud (IONOS → api.kimi.com). Format OpenAI natif (pas de conversion Anthropic nécessaire).

Comportement parallel tool calls de Kimi K2

Kimi K2 peut émettre plusieurs tool_calls dans une seule réponse — c'est sa feature "parallel agent deployment". Exemple :

{
  "choices": [{
    "message": {
      "tool_calls": [
        {"id": "call_1", "function": {"name": "system__bash", "arguments": "{\"command\":\"ls /var/log\"}"}},
        {"id": "call_2", "function": {"name": "system__python", "arguments": "{\"code\":\"import os; print(os.listdir('.'))\"}"}},
        {"id": "call_3", "function": {"name": "system__web_search", "arguments": "{\"query\":\"nginx error 502\"}"}}
      ]
    },
    "finish_reason": "tool_calls"
  }]
}

Ces 3 tools sont exécutés simultanément par ThreadPoolExecutor(max_workers=4). C'est la "parallelisation" observée dans les démos Kimi K2.

Reasoning content

Kimi K2 inclut un champ reasoning_content dans ses réponses streaming — sa réflexion interne. ClawbotCore l'accumule dans reasoning_acc mais ne l'affiche pas encore dans le dashboard (il est conservé pour le round-trip de context).

Limites actuelles

L1
Un seul agent par requête (auto-routing)

Le routeur Haiku ne sélectionne qu'un agent (matched[:1]). Même si la tâche nécessiterait Python Dev + SysAdmin en parallèle, seul le premier est appelé. Le routing multi-agent parallèle nécessite un agent_ids explicite depuis le frontend.

L2
Twins Partner utilise Anthropic direct (pas le cloud)

Le reviewer _twins_review() appelle api.anthropic.com/v1/messages directement avec la clé Anthropic locale. Si la clé n'est pas dans l'env du Pi, le review échoue silencieusement.

L3
Reasoning content Kimi non affiché

Le champ reasoning_content de Kimi K2 est accumulé côté orchestrateur mais pas rendu dans le dashboard. L'utilisateur ne voit pas la réflexion interne de Kimi.

L4
Compaction auto peut interrompre le mode agent

Quand COMPACT_THRESHOLD = 15000 est atteint, Haiku résume la conversation via /v1/chat/completions (Core mode), ignorant le contexte agent. Le résumé perd la notion d'agent actif.

L5
MAX_TOOL_ROUNDS = 15 partagé entre tous les agents

En mode multi-agent parallèle, chaque agent a son propre compteur. Mais en séquentiel (un seul agent), la limite de 15 rounds peut bloquer des tâches complexes avant complétion.

Pistes d'amélioration

A1
Routeur multi-agent — sélectionner plusieurs agents en une passe Facile

Modifier _route_via_llm() pour retourner une liste d'IDs au lieu d'un seul. Haiku décide "cette tâche nécessite python-dev ET sysadmin". Changer matched[:1] → matched[:3] dans main.py.

# Prompt modifié :
"Réponds avec les IDs séparés par virgule (ex: python-dev,sysadmin) ou: none"
# Parsing :
chosen_ids = [x.strip() for x in chosen.split(",") if x.strip() in enabled]
A2
Afficher le reasoning_content de Kimi dans le dashboard Facile

Ajouter un event SSE reasoning depuis _call_llm_stream() quand reasoning_acc est non vide. Le dashboard l'affiche en italique/gris avant la réponse (bouton Trace activé).

A3
Routeur KimiCode — laisser le modèle sélectionné router Moyen

Passer routing_model = data.get("model", "claude-haiku-4-5-20251001") à route_to_agents() puis à _route_via_llm(). Si KimiCode sélectionné : Kimi classe les agents. Avantage : Kimi K2 est meilleur que Haiku pour la planification de tâches coding. Coût : légèrement plus cher pour le routing.

A4
Agent "Planner" dédié — décompose en sous-tâches parallèles Moyen

Créer un agent Planner avec un prompt qui force la décomposition : "Décompose cette tâche en sous-tâches indépendantes, retourne un JSON [{agent_id, subtask}]". Le résultat est parsé et lance plusieurs chat_with_agent_stream() en parallèle.

# orchestrator.py
def chat_with_planner_agents_stream(request_body):
    plan = _call_planner(request_body)  # retourne [{agent_id, task}]
    agent_ids = [p["agent_id"] for p in plan]
    yield from chat_with_multi_agents_stream(request_body, agent_ids)
A5
Handoff inter-agents — un agent passe la main Complexe

Outil system__handoff déjà prévu dans le code (skill system__handoff injecte la liste des agents disponibles). Implémenter l'outil réellement : quand appelé, spawn un nouveau chat_with_agent_stream() avec le contexte actuel. Attention aux boucles (agent A passe à B qui repasse à A).

A6
Mémoire partagée inter-agents Complexe

Actuellement chaque agent reçoit le même request_body. Pas de mémoire partagée entre agents parallèles. Implémenter un blackboard (dict partagé accessible via outil system__memory_set/get) pour que SysAdmin partage ses findings avec Python Dev.

Table de décisions

SituationMode recommandéModèle recommandéPourquoi
Question simple, réponse rapide ClawBot Core Haiku Pas besoin de routing, pas d'agent overhead
Debug système Pi (services, logs) Core-Agent Haiku ou Sonnet SysAdmin agent avec bash/ssh skills
Écrire / refactorer du code Core-Agent KimiCode 2.5 262k ctx, parallel tools natifs, entraîné coding
Tâche multi-domaines (code + système + recherche) Core-Agent KimiCode 2.5 ou Opus Kimi spawne plusieurs tool_calls en parallèle, Opus plan mieux
Analyse long document / contexte étendu ClawBot Core KimiCode 2.5 262k ctx vs 200k Anthropic
Architecture / décisions stratégiques ClawBot Core Opus Raisonnement le plus profond
Production : économiser tokens Core-Agent KimiCode 2.5 (plan Free) Budget model, alloué aux plans Free en fallback

Résumé : Pour du coding agentic parallèle, Core-Agent + KimiCode 2.5 est le setup optimal. Kimi K2 émet plusieurs tool_calls par tour, l'orchestrateur les exécute en parallèle, 262k de contexte. Le routeur Haiku (30 tokens, ~100ms) sélectionne le bon agent sans impacter les performances.
