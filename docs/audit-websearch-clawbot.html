⚡ Verdict synthétique
Le système web search ClawBot est le meilleur disponible pour la contrainte hardware Pi H3 1GB + stdlib-only. Son architecture 3 couches (scraping → API AI → signal Alia), son self-healing autonome, sa gestion CN native et son modèle de sécurité (zéro clé sur le device) le positionnent au-dessus d'OpenClaw et PicoClaw sans comparaison possible.

Les principaux gaps vs l'état de l'art commercial sont : l'absence de Tavily (source RAG haute qualité) et l'absence de cache requêtes. Ces deux éléments sont des quick wins à moins de 2 jours de développement.

Self-healing patterns — pipeline
Mécanisme autonome de réparation quand un moteur change son HTML.
_adapt_engine_patterns(engine_name, engine_cfg, html_sample, user_model) 1. Fetch HTML du moteur → 0 résultats avec patterns actuels 2. Envoyer 5000 chars HTML brut + patterns échoués au LLM actif 3. LLM retourne nouveau JSON de patterns (block/ddg/brave_json) 4. Retry _parse_results() avec nouveaux patterns 5. ✓ Succès → sauvegarder dans search-engines.json → reliability: min(1.0, r*0.9+0.1) (+10%) → retourner "Search results via [engine — AI repaired] for: ..." 6. ✗ Échec → reliability: max(0.1, r*0.8) (-20%) → moteur suivant dans la boucle

vs PicoClaw
Go agent Sipeed v0.2.0
PicoClaw n'a aucun système de recherche web propre. Il délègue 100% au LLM configuré. ClawBot le surpasse totalement sur ce point. PicoClaw est un orchestrateur agent, pas un moteur search.

ClawBot +++ supérieur
🆚
vs OpenClaw
Variante sans cloud
OpenClaw manque : self-healing, UA rotation, cloud proxy sécurisé, gestion CN, tool injection dynamique. Acceptable pour usage solo dev, insuffisant pour un produit multi-utilisateurs.

ClawBot ++ supérieur
🆚
vs Kimi / Claude Search natifs
API directe sans proxy
Même qualité de résultats. La valeur ajoutée ClawBot : les clés ne quittent pas le cloud, fallback scraping gratuit, quota géré côté plateforme, 2-round Kimi géré automatiquement.

ClawBot ≈ + proxy sécurisé
🆚
vs SearXNG
Méta-moteur open source
SearXNG agrège 70+ moteurs — supérieur en diversité. Mais impossible sur Pi H3 1GB (Docker, 512MB+ RAM). ClawBot fait le bon compromis stdlib pour hardware contraint.

ClawBot + sur hardware embarqué
🆚
vs Tavily
API LLM-optimisée
Tavily est RAG-ready, extraits complets, $0/1000 searches (free tier). Absent de la stack ClawBot. C'est le gap le plus impactant — à ajouter comme 4ème outil system__web_search_tavily.

Tavily + pour RAG — à intégrer
🆚
vs Open WebUI
Perplexica / SearXNG intégré
Open WebUI nécessite Docker multi-services et 1GB+ RAM. Incompatible Pi H3. ClawBot cible exactement ce que Open WebUI ne peut pas faire : IA embarquée sur hardware minimal.

Avantages compétitifs validés en production.
⚡
Zéro dépendance
stdlib Python uniquement
  • Fonctionne sur Pi H3 1GB sans pip install
  • Pas de Docker, pas de services externes requis
  • Démarrage instantané, <10MB RAM utilisés
  • Aucune surface d'attaque de supply chain
🔄
Graceful degradation 3 couches
Jamais de "no results" sans effort
  • Scraping local → API AI → signal Alia
  • Chaque couche tente en séquence
  • Alia appelle elle-même le bon outil en Auto
  • Le système ne tombe jamais silencieusement
🧠
Self-healing autonome
Réparation sans intervention humaine
  • HTML 0 résultats → LLM génère nouveaux regex
  • Patterns sauvegardés dans search-engines.json
  • Scoring dynamique : fiables montent, défaillants descendent
  • LLM adapte aussi sa requête (détection hors-sujet)
🔒
Sécurité by design
Zéro clé API sur le device
  • Pi ne connaît que son sub_key
  • Clés Moonshot/Anthropic sur cloud seulement
  • Device compromis = aucune clé exposée
  • Révocation sub_key côté cloud instantanée
🇨🇳
Couverture CN native
Meilleur du marché sur Pi
  • Auto-détection IP → pool CN si GFW détecté
  • Baidu, Bing CN, Sogou intégrés
  • Moteurs GFW-bloqués exclus automatiquement
  • Traduction FR→ZH via LLM pour moteurs CN
🎛️
Tool injection dynamique
Contrôle fin sans surcoût token
  • Pi / LLM First / Auto enforced au niveau payload API
  • Descriptions dans tools[], pas dans system prompt
  • Mode Pi = Kimi/Claude physiquement absents du payload
  • Auto = Alia décide selon qualité, pas selon provider

Observations critiques issues des tests
Comportements non documentés avant les tests live, maintenant connus.
⚠️
Anomalie #1 — KimiCode 2.5
Hallucination nom d'outil
KimiCode 2.5 appelle web__search (double underscore, sans system__ prefix) au lieu de system__web_search. L'orchestrateur accepte car _web_search() est le handler interne. Risque : si le modèle hallucine un nom totalement inconnu → erreur tool_not_found silencieuse. Fix : alias dans le dispatcher.
💡
Observation #2 — Cross-provider Auto
Haiku choisit Kimi search
En mode Auto, Haiku a choisi system__web_search_kimi (pas Claude) pour son escalade finale. Confirme l'architecture : les descriptions d'outils neutres/égales laissent le LLM décider selon la qualité attendue, indépendamment de son propre provider. Comportement optimal.
🔴
Observation #3 — Bing CN hors-sujet
Requêtes françaises vs index CN
Bing CN retourne des résultats de jeux vidéo japonais pour "dernière version Python 2024". Le LLM (Haiku) a détecté l'anomalie et relancé en anglais — self-healing LLM niveau requête, distinct du self-healing patterns HTML. À gérer aussi côté code (filtre langue résultats).
